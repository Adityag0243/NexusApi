What You Are Building 
NexusAPI is a backend platform with the following shape. Read this section completely before writing any code. 
Component 
	What it does
	Organisations 
	The top-level tenant. Every user belongs to one organisation. Data from one organisation must never be visible to another. This is enforced at the database query level, not just in application code.
	Users 
	Belong to an organisation. Authenticate via Google OAuth. Receive a signed JWT token on login. All protected endpoints validate this token on every request.
	Credit ledger 
	Each organisation has a credit balance. Credits are tracked via a transaction log — every addition and every deduction is recorded as a row, never as a direct balance update. The current balance is derived from the ledger.
	Product endpoints Background job 
	Two endpoints that represent AI-powered features. Each costs a fixed number of credits per call. Access is blocked if the organisation does not have sufficient credits. The deduction must be atomic. 
One of the two product endpoints must process its work asynchronously. It returns a job ID immediately. The caller polls a separate endpoint for the result.

Failure Modes — Read This Section 
This is not optional 
Handling failure modes correctly carries the same weight as functional correctness in how your submission is evaluated. 
A system that works perfectly when everything goes right but crashes or produces wrong results when things go wrong is not production-ready. Every scenario below must be handled explicitly. 
For every scenario listed below, your system must respond correctly. ‘Correctly’ means: the right HTTP status code, a structured error response, no data corruption, and the system continues operating normally afterwards. 
Kasparro AI — Backend Engineering
Authentication failures 
• JWT token is missing from the request 
• JWT token is present but expired 
• JWT token is present, not expired, but the signature has been tampered with • JWT token is valid but the user has been removed from the organisation (user_id no longer exists in the database) 
Credit system failures 
• POST /api/analyse is called with exactly 0 credits remaining 
• POST /api/analyse is called with 24 credits remaining (one less than required) • Two simultaneous requests arrive for the same organisation, each requiring 25 credits, and the organisation has exactly 25 credits total — only one should succeed • POST /api/summarise is called, the job is queued, but the background worker fails before processing completes 
Input validation failures 
• POST /api/analyse is called with text shorter than 10 characters 
• POST /api/analyse is called with text longer than 2000 characters 
• POST /api/analyse is called with a missing text field entirely 
• GET /api/jobs/{job_id} is called with a job_id that does not exist 
• GET /api/jobs/{job_id} is called with a valid job_id that belongs to a different organisation 
Infrastructure failures 
• The database is temporarily unreachable — what does /health return? • Redis is temporarily unreachable — does the API keep working or go down entirely? • A background job is still pending after 5 minutes — what is the job status and what happens to the credits?

The five questions you must answer 
Question 1 
Why does the credit system use a transaction ledger instead of a balance column? 
Explain the design decision behind CreditTransactions. What problem does a ledger solve that a simple balance column does not? What are the tradeoffs? 
Question 2 
How did you handle the simultaneous credit deduction problem? 
Two requests arrive at exactly the same time. The organisation has exactly 25 credits. Both requests cost 25 credits. Walk through exactly what happens in your implementation. What database mechanism prevents both from succeeding? Why did you choose that mechanism over alternatives? 
Question 3 
What happens when the background worker fails after credits have been deducted? 
The job was queued. Credits were taken. The worker crashed before completing. Describe what your system does in this scenario. Did you implement a refund? A retry? Something else? Why? 
Question 4 
How does your idempotency implementation work, and where does it live? 
Walk through the flow of a duplicate request with the same Idempotency-Key. Where is the check performed? What database constraint backs it up? What happens if two identical requests arrive simultaneously — before either has completed? 
Question 5 
What would break first at 10x the current load, and what would you do about it? 
Assume your system is handling 10 times as many requests. You don’t need to implement anything. Just identify the most likely bottleneck and describe in one paragraph what you would do about it.
